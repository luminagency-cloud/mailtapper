0Phase 0 — Dogfood (build for yourself first)

v1's first user and consumer is the founder, dogfooding through luminmail — the existing webmail app, repurposed as a thin test client (one-line swap of its sync to GET /v1/messages). That real, in-use path is the v1 validation: if luminmail reliably reads mail through mailtapper, the approach is proven.

The larger AI life-planning app is the motivating use case but is NOT built yet — it becomes a consumer later, not a v1 dependency.

External market validation (the 5-qualified-yeses community test) is deferred to v2 — relevant only when deciding to productize mailtapper for other people. The Validation Test Kit stays parked. For now, adoption by others is an explicit non-goal — the only success metric is "luminmail reads my mail through mailtapper."

1v1 — Committed scope

Identity: an access layer / fetch service. Not a mail client, not a store. The mailbox is the source of truth; the caller owns durable data.

  • IMAP only, read-only.
  • Model: access layer + thin cache — live fetch on demand; connection hygiene (per-host connection reuse, concurrency cap, retry + backoff); short-TTL cache. No long-term mail storage.
  • Control plane: register/manage connections — host, port, explicit TLS mode (ssl / starttls / none), allow-invalid-cert, username, password. Validated before save, encrypted at rest.
  • Data plane: query normalized mail by account, sender, time window, unread, full-text → one unified JSON shape.
  • Unified message: content-hash internal_id (caller dedupe) + opaque id locator (live single-message re-fetch), provider tag, has_attachments + metadata, clean HTML-stripped body_text.
  • Security: per-tenant API keys (hashed, scoped); envelope encryption (KEK→DEK, AAD, key-version rotation path).
  • Scope: INBOX first.
  • Stack: Node/TS · Fastify · ImapFlow · Postgres (Neon now) · Docker + Caddy.
  • Human surface: API-first, plus a thin connection-manager view.

2Cost & abuse guardrails (v1)

Principle: mailbox size is not your cost — query breadth is. Connecting a 30k-message mailbox costs ~nothing (we don't slurp or store it). Cost is per-fetch compute + egress, only on what's fetched. So bound fetches, not mailbox size. No LLM cost — the caller pays for its own AI.

Guardrails (cheap, structural — build even for a single user): no unbounded fetch (enforced pagination + default recent window); per-tier lookback cap; per-key rate limits + per-tenant concurrency cap; per-tier connection cap; thin cache absorbs repeats. Heavy-user escape valve = self-host, not BYOK.

4v2 & directional

  • Optional longer per-tenant retention — the "memory access layer" option: let callers have mailtapper hold a window of mail, not just pass through. Opt-in, never default.
  • KMS-hosted KEK + automated periodic key rotation + per-key IP allowlists.
  • Connection diagnostic tool — probe a mailbox for TLS/folder/UID quirks. Support tooling and the long-tail moat.
  • Distribution: OpenAPI docs, an n8n node / Make module, client SDKs.
  • Multi-worker scale-out — once push/polling load justifies it.
  • External market validation & GTM — the 5-yes community test + Validation Test Kit, when productizing for others.
  • Pricing productization — per-connection base + volume tiers.
