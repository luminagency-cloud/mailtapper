🛠️ mailtapper — Build Spec (v1)

An access layer for email. Canonical v1 build spec — access-layer model, live fetch, no stored mail.

Portable snapshot. Authoritative working copy lives in the project workspace. Last synced 2026‑06‑21.

Overview & Principles

Mailtapper is an access layer for email: connect any IMAP mailbox and read it as clean, unified JSON through one API. Built for AI agents and automations, not humans. This is the v1 build spec.

Model: access layer (live fetch + thin cache). Mailtapper fetches from the mailbox on demand, normalizes, and returns — it does not store mail. The mailbox is the source of truth; the caller owns any durable copy. A short-TTL cache and a connection-hygiene layer make live fetch fast and resilient.

Principles

  • Fetch service, not a store. No mail database, no retention. Minimal liability: we hold credentials, not contents.
  • Bounded by default. No "give me everything" call — every query is windowed, paginated, and capped.
  • Multi-tenant from row one, per-tenant API keys.
  • Built to hold credentials safely: explicit per-connection TLS, envelope-encrypted secrets.

Stack: Node 20 + TypeScript, Fastify, ImapFlow, mailparser + html-to-text, Postgres (Neon now; vanilla SQL so it ports to a local container), Docker + Caddy on the Ubuntu host.

Not in v1: push/webhooks, POP3, multi-folder, attachment bytes, sending, OAuth providers (Gmail/M365), self-host, metered billing — see the Roadmap.

Data Model

No messages or sync_state tables — we don't store mail. The DB holds config + credentials only.

CREATE TABLE tenants (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  name text NOT NULL,
  tier text NOT NULL DEFAULT 'free',
  created_at timestamptz NOT NULL DEFAULT now()
);

CREATE TABLE api_keys (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id uuid NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
  key_prefix text NOT NULL,            -- e.g. 'mtapper_live_AbCd1234'
  key_hash   text NOT NULL,            -- sha256(full key)
  scopes     text[] NOT NULL DEFAULT '{messages:read}',
  last_used_at timestamptz, last_used_ip inet,
  created_at timestamptz NOT NULL DEFAULT now(), revoked_at timestamptz,
  UNIQUE (key_prefix)
);

CREATE TABLE tenant_keys (             -- envelope encryption (see Credential Security)
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id uuid NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
  dek_wrapped bytea NOT NULL, dek_iv bytea NOT NULL,
  kek_version int NOT NULL, dek_version int NOT NULL,
  retired_at timestamptz, created_at timestamptz NOT NULL DEFAULT now()
);

CREATE TABLE connections (
  id uuid PRIMARY KEY DEFAULT gen_random_uuid(),
  tenant_id uuid NOT NULL REFERENCES tenants(id) ON DELETE CASCADE,
  label text,
  protocol text NOT NULL DEFAULT 'imap' CHECK (protocol IN ('imap')),  -- pop3 later
  host text NOT NULL, port integer NOT NULL,
  tls_mode text NOT NULL DEFAULT 'ssl' CHECK (tls_mode IN ('ssl','starttls','none')),
  allow_invalid_cert boolean NOT NULL DEFAULT false,
  username text NOT NULL,
  secret_ciphertext bytea NOT NULL, secret_iv bytea NOT NULL,
  secret_dek_id uuid REFERENCES tenant_keys(id),
  provider text NOT NULL DEFAULT 'imap_generic',
  source_account text NOT NULL,
  status text NOT NULL DEFAULT 'pending' CHECK (status IN ('pending','active','error','paused')),
  last_error text, last_validated_at timestamptz,
  created_at timestamptz NOT NULL DEFAULT now(), updated_at timestamptz NOT NULL DEFAULT now(),
  UNIQUE (tenant_id, username, host)
);

The thin cache is in-memory (single-node) in v1 — no table. Redis/Postgres-backed cache only if/when we run multiple api nodes.

The Unified Message (contract)

Two ids, two jobs:

  • id — an opaque locator encoding connection_id + folder + UID + uidvalidity, HMAC-signed. Use it for GET /v1/messages/{id} to re-fetch that exact message live. A stale locator (UIDs reset) returns 410 Gone (re-query).
  • internal_id — a content hash (RFC Message-ID, else a from + date + subject + body fingerprint). For your dedupe across your store. We compute it on the fly; we don't store it.
{
  "id": "b64url.signed.locator",
  "internal_id": "sha256...",
  "connection_id": "uuid",
  "source_account": "you@yourdomain.com",
  "provider": "imap_generic",
  "folder": "INBOX",
  "message_id": "<abc@host>",
  "thread_id": "...",
  "from": { "name": "Jane", "email": "jane@example.com" },
  "to": [], "cc": [],
  "subject": "Project update",
  "received_at": "2026-06-21T11:30:00Z",
  "body_text": "Clean, HTML-stripped text...",
  "has_attachments": true,
  "attachment_count": 2,
  "attachments": [ { "filename": "po.pdf", "content_type": "application/pdf", "size_bytes": 18234 } ],
  "is_unread": true,

Body: prefer the plain-text MIME part; else convert HTML to text. Raw HTML never lands in body_text. Attachments: metadata only in v1 (no bytes).

Data-Plane API (live fetch)

GET /v1/messages — live fetch, bounded. Filters map to IMAP SEARCH: since/until -> SINCE/BEFORE, from -> FROM, unread -> UNSEEN, subject -> SUBJECT, connection_id (one; omit = all the tenant's active connections), limit (default 25, tier max), cursor (keyset).

1. authz: api key -> tenant; enforce rate limit; clamp limit to tier max
2. clamp since to tier lookback cap (older -> 403 + upgrade hint)
3. cache hit? (tenant + normalized query) -> return
4. for each target connection (bounded by per-tenant concurrency cap), on a pooled IMAP conn:
     SEARCH(filters) -> matching UIDs (newest first)
     take page window after cursor -> FETCH headers+body -> normalize
5. merge across connections by received_at desc; apply limit
6. cache (short TTL); return { data: [...], next_cursor }

Pagination: keyset by (received_at, uid); next_cursor encodes position. Cross-account queries fan out per connection and merge — the heaviest call, which is exactly what the concurrency + connection caps bound.

GET /v1/messages/{id} — decode locator -> authorize against the caller's tenant -> live FETCH that UID -> normalize. Stale uidvalidity -> 410.

Search honesty (a real model trade-off): with no stored index, search is whatever IMAP SEARCH supports. since/until/from/unread/subject are reliable everywhere. Full-body text (q) maps to IMAP TEXT/BODY, which is server-dependent and can be slow — so v1 offers it best-effort and documents it may be unsupported on some hosts. Fast, rich cross-account full-text is the main capability we trade away by not storing; an opt-in stored-index tier could add it back later.

Limits & Guardrails

Principle: mailbox size is not your cost — query breadth is. Connecting a 30k-message mailbox costs ~nothing (we don't slurp it); cost is per-fetch compute/egress, only on what's fetched. So we bound fetches, not mailbox size. No LLM cost — the caller pays for its own AI.

Enforcement: defaults (last 7 days, 25/page) apply when params are omitted, regardless of tier — the tier sets the ceiling. Rate hit -> 429 + Retry-After. Lookback beyond tier -> 403 + upgrade hint. Page over max -> clamp. Connection cap -> blocked at creation with upgrade prompt. Heavy-user valve = self-host (not BYOK — there's no per-request vendor key; the cost is compute). Future metered dimension: messages fetched/month (deferred with billing).

Connector & Normalizer

Validator (lifted from luminmail, fixed — explicit TLS, IMAP-only, timeouts):

const client = new ImapFlow({
  host: c.host, port: c.port,
  secure: c.tlsMode === "ssl",                 // explicit -- never derived from the port
  tls: { rejectUnauthorized: !c.allowInvalidCert },
  connectionTimeout: 10_000, greetingTimeout: 7_000, socketTimeout: 30_000,
  auth: { user: c.username, pass }, logger: false,
});

formatMailError keeps luminmail's buckets (auth/timeout/DNS/TLS) + adds econnrefused and self-signed-cert ("-> enable allow_invalid_cert").

Normalizer (runs in the request path): simpleParser(raw) -> prefer .text, else html-to-text(.html); map addresses; detect attachments -> metadata; compute internal_id; build the signed id locator; derive thread_id. HTML-strip + attachment detection are the two things luminmail never did.

internal_id: messageId ? "mid:" + lower(messageId) : "fp:" + sha256(from|dateISO|subject|sha256(bodyText)), then sha256(connection_id + "\n" + that).

Credential Security

We hold mailbox credentials, not contents — so this is the crown jewel.

  • Envelope encryption (KEK -> DEK). Master KEK (32 random bytes) lives outside the DB — env MAILTAPPER_KEK_B64 now, KMS later; loaded as a versioned keyring. Per-tenant DEK (random 32 bytes) encrypts secrets and is stored wrapped by the KEK in tenant_keys.
  • Cipher: AES-256-GCM, random 12-byte IV, AAD = conn:<connection_id> so ciphertext can't be transplanted between rows/tenants.
  • Rotation: kek_version / dek_version columns + keyring let old and new coexist — rotate the KEK by re-wrapping DEKs; rotate a DEK by re-encrypting one tenant's secrets. No big-bang.
  • API keys: mtapper_live_<32 random bytes>; store sha256(key) + prefix; verify constant-time; scoped (messages:read for read-only callers like luminmail). Shown once.
  • Point-of-use only: secrets decrypted in-memory at fetch time, never logged or persisted; deleting a connection wipes its secret.

Connection Hygiene & Cache

The layer that makes live fetch fast and survivable on flaky long-tail hosts (and answers the "12 timeouts in a row" worry).

  • Connection pool: reuse ImapFlow connections per (host, username) instead of reconnecting each call.
  • Concurrency caps: per host and per tenant (tier-driven) — never open a storm of sockets to one server (that's what causes the timeouts/bans).
  • Retry with backoff + jitter: on transient errors, a few retries with growing delay — not 12 instant hammers. Hard failures surface as a clear error + set the connection status='error'.
  • Timeouts on connect/greeting/socket so a dead host fails fast.
  • Thin cache: in-memory LRU keyed by (tenant, normalized-query) and by (connection, uid) for single messages; short TTL (default ~60s, configurable). Wipeable at any instant with zero data loss — the mailbox is always the source of truth. It is a performance buffer, not storage.

Deployment

All on the Ubuntu/Docker host. No worker in v1.

services:
  api:
    build: .
    command: node dist/api.js
    environment:
      DATABASE_URL: ${DATABASE_URL}                 # Neon (vanilla PG -> portable to local later)
      MAILTAPPER_KEK_B64: ${MAILTAPPER_KEK_B64}      # 32 random bytes, base64 (openssl rand -base64 32)
      MAILTAPPER_LOCATOR_SECRET: ${LOCATOR_SECRET}   # HMAC key for the message id locator
      PUBLIC_BASE_URL: https://api.yourdomain.com
    expose: ["3000"]
  caddy:
    image: caddy:2
    ports: ["80:80","443:443"]
    volumes: ["./Caddyfile:/etc/caddy/Caddyfile","caddy_data:/data"]
volumes: { caddy_data: {} }

Migrations via Drizzle on deploy. Moving Neon -> local Postgres later is a DATABASE_URL swap (vanilla SQL, no Neon-specific features).

Scope & Deferred

v1 (this spec): IMAP, read-only, pull-only live fetch + thin cache, INBOX, bounded/paginated queries with tier caps, per-tenant scoped API keys, envelope-encrypted credentials, connection-hygiene layer. DB = config + keys only.

Decided: pull-only (no push) for v1. Callers poll the data plane on their own cadence; mailtapper holds no per-connection cursor and runs no background worker. Rationale: the ICP (polling agents/automations) doesn't need real-time delivery; push would reintroduce the worker + persistent cursor + delivery infra the access-layer model deliberately removed, and would invert the cost profile to continuous background IMAP polling. "Opt-in" would limit runtime load but not build scope. Deferring is free: push is purely additive (the read contract is unchanged).

Key trade-off accepted: no stored full-text index -> search limited to IMAP SEARCH (reliable on headers/since/from/unread/subject; full-body best-effort).

Deferred (Roadmap): push/webhooks (v3 candidate, demand-gated), POP3, multi-folder, attachment bytes, Microsoft 365 & Gmail (OAuth), sending, self-host, metered billing, optional stored-index/cache tier for rich search & longer retention.

Open decisions still live: pricing model.
